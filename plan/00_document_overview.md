# 00. 문서 개요

## 1. 문서 목적

이 문서는 자산 운용 로그라이트 게임의 MVP 시스템 기획서이자 Unity 구현을 위한 기능 명세 초안이다.

본 문서의 목적은 다음과 같다.

1. 게임의 핵심 시스템 구조를 명확히 정의한다.
2. Unity 구현 시 필요한 화면 상태, 데이터 구조, 처리 순서, 예외 규칙을 정리한다.
3. Codex 또는 개발자가 기능 단위로 구현할 수 있도록 각 시스템의 책임 범위를 분리한다.
4. 이미 확정된 기획 결정을 기준으로, 불필요한 재논의를 줄이고 구현 가능한 명세로 전환한다.

본 문서는 밸런스 최종 확정 문서가 아니다.
카드별 비용, 운용가치, 운용 수익, 분기 목표 수익, 평가 기준값 등은 테이블 기반으로 조정 가능하게 둔다.

---

## 2. 대상 독자

이 문서는 다음 독자를 대상으로 한다.

- 게임 디자이너
- Unity 클라이언트 개발자
- UI/UX 구현 담당자
- 데이터 테이블 설계 담당자
- Codex 기반 구현 에이전트

특히 Codex가 각 시스템을 순차적으로 구현할 수 있도록, 문서의 각 장은 가능한 한 다음 구조를 따른다.

```text
- 시스템 목적
- 핵심 규칙
- 화면/상태 전환
- 처리 순서
- 예외 처리
- 구현 메모
- 데이터 구조 제안
```

## 3. 문서 범위

본 기획서는 MVP 구현에 필요한 핵심 시스템을 다룬다.

포함 범위는 다음과 같다.

```text
- 런 구조
- 영업일 / 분기 / 회계년도 시간 구조
- 시장 영역 상태
- 시장 테이프
- 자원 구조
- 자산 카드 구조
- 카드 상세보기
- 자산 매수
- 칩 조작 UX
- 딜
- 예약 시스템
- 자원 확보
- 추가 매수권
- 분기 마감
- 환매 압력
- 런 실패
- 4Q 휴가
- 최종 정산
- 최종 평가
- 데이터 테이블 구조
- MVP 범위
```

포함하지 않는 범위는 다음과 같다.

```text
- 최종 밸런스 수치 확정

- 아트 리소스 상세 스펙
- 사운드 상세 스펙
- 고급 연출 디테일
- 튜토리얼 전체 시나리오
- 라이브 밸런싱 정책
```

## 4. MVP 기준

MVP는 게임의 핵심 판단 구조와 한 런의 진행이 끝까지 작동하는 것을 목표로 한다.

MVP에서 반드시 포함해야 하는 요소는 다음과 같다.

```text
- 영업일 진행
- 분기 진행
- 회계년도 진행
- 시장 테이프 표시 및 진행
- 카드 상세보기
- 자산 매수
- 예약
- 딜 사용
- 자원 확보
- 전문 자원 한도
- 분기 마감 정산
- 환매 압력 증가
- 환매 압력 한도에 의한 런 실패
- 최종 정산
- 최종 평가
```

MVP에서 후순위로 둘 수 있는 요소는 다음과 같다.

```text
- 복잡한 연출
- 장문의 튜토리얼
- 고급 애니메이션
- 휴가 이벤트
- 추가 보상 선택지
- 세밀한 통계 리포트
- 카드별 특수 연출
```

MVP의 핵심 목표는 “게임이 처음부터 끝까지 규칙적으로 돌아가는 것”이다.
시각적 완성도와 고급 피드백은 이후 단계에서 보강한다.

## 5. 핵심 용어

본 문서에서는 다음 용어를 사용한다.

| 용어 | 의미 |
|---|---|
| 영업일 | 기존 턴에 해당하는 최소 행동 단위 |
| 다음 영업일 | 현재 영업일을 종료하고 다음 영업일로 넘어가는 입력 |
| 분기 | 기존 스테이지에 해당하는 중간 진행 단위 |
| 회계년도 | 여러 분기로 구성된 상위 진행 단위 |
| 운용가치 | 자산 카드가 가진 고유 점수. 스플렌더의 승점과 같은 개념 |
| 운용 수익 | 보유 자산의 영업일 시작 현금, 분기 마감 정산 등 운용 성과로 얻은 현금 |
| 조달 현금 | 자원 확보로 얻은 현금. 운용 수익에는 포함하지 않음 |
| 현금 | 자산 매수에 사용되는 기본 결제 자원 |
| 전문 자원 | 리서치, 신용, 원자재 |
| 딜 | 전문 자원을 대체하고 현금 비용을 낮추는 예약 보상 자원 |
| 환매 압력 | 누적 리스크 수치. 한도에 도달하면 런 실패 |
| 시장 테이프 | 매도 임박 / 현재 시장 / 예비 시장으로 구성된 카드 흐름 |
| 시장 테이프 진행 | 카드들이 한 단계씩 밀려나는 처리 |
| 시장 테이프 갱신 | 시장 테이프 전체를 새 카드로 재구성하는 처리 |
| 예약 카드 | 시장에서 빼내어 예약 구역에 장기 보관한 카드 |
| 보유 자산 | 매수 완료되어 포트폴리오에 추가된 카드 |

## 6. 용어 사용 원칙

### 6.1 AUM 용어 사용 금지

이 게임에서는 `AUM`이라는 용어를 사용하지 않는다.

대신 다음 표현을 사용한다.

| 사용 금지 | 사용 |
|---|---|
| AUM | 운용가치 |
| 최종 AUM | 최종 운용가치 |
| 카드 AUM | 카드 운용가치 |
| AUM 기준 평가 | 운용가치 기준 평가 |

운용가치는 현실 금융의 총 운용자산 개념이 아니라, 자산 카드가 가진 고유 점수이다.
현금, 리서치, 신용, 원자재, 딜은 운용가치에 포함되지 않는다.

```text
최종 운용가치 = 보유 자산 카드들의 운용가치 합계
```

---

### 6.2 턴 용어 사용 금지

유저-facing 텍스트에서는 `턴`이라는 표현을 사용하지 않는다.

| 사용 금지 | 사용 |
|---|---|
| 턴 | 영업일 |
| 턴 종료 | 영업일 종료 |
| 턴 넘기기 | 다음 영업일 |
| 남은 턴 | 남은 영업일 |

내부 코드에서는 필요에 따라 `Turn`이라는 명칭을 사용할 수 있으나, UI와 기획 문서에서는 `영업일`을 기준으로 작성한다.

---

### 6.3 스테이지 용어 사용 금지

유저-facing 텍스트에서는 `스테이지` 대신 `분기`를 사용한다.

| 사용 금지 | 사용 |
|---|---|
| 스테이지 | 분기 |
| 스테이지 목표 | 분기 목표 |
| 스테이지 마감 | 분기 마감 |
| 스테이지당 턴 수 | 분기당 영업일 수 |

## 7. 구현 우선순위 원칙

구현 시 다음 우선순위를 따른다.

1. 데이터 구조와 상태 전환을 먼저 구현한다.
2. 기능이 규칙대로 동작하는 것을 우선한다.
3. UI 배치는 Unity 엔진 작업 단계에서 조정한다.
4. 연출은 최소 구현 후 단계적으로 보강한다.
5. 밸런스 수치는 테이블화하여 조정 가능하게 둔다.

특히 버튼의 정확한 위치, 크기, 색상, 레이아웃은 본 문서에서 강제하지 않는다.
본 문서는 어떤 입력이 존재하고, 어떤 상태에서 활성화되며, 클릭 시 어떤 처리가 일어나는지를 정의한다.

## 9. 확정된 큰 구조 요약

게임의 최상위 구조는 다음과 같다.

```text
영업일 < 분기 < 회계년도
```

회계년도 구성은 다음과 같다.

```text
1회계년도
- 1Q: 플레이
- 2Q: 플레이
- 3Q: 플레이
- 4Q: 휴가

2회계년도
- 1Q: 플레이
- 2Q: 플레이
- 3Q: 플레이
- 4Q: 휴가

3회계년도
- 1Q: 플레이
- 2Q: 플레이
- 3Q: 플레이
- 4Q: 플레이
```

영업일 수는 다음과 같다.

```text
1회계년도: 분기당 4영업일
2회계년도: 분기당 4영업일
3회계년도: 분기당 5영업일
```

총 플레이 분기 수는 10개이며, 총 플레이 영업일 수는 44일이다.
